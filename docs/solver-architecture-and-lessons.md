# Maze Trials Solver: 完整架构与踩坑记录

## 1. 问题定义

### 游戏规则

两个六足机器人（R1、R2）在网格迷宫中协作，将昏迷队友 R3（装在红色货箱里）运送到安全区的 Repair Room。

- **推（Push）**：机器人走进箱子，箱子向前移一格（前方必须为空）
- **拉（Tow）**：机器人挂接箱子后，沿直线远离箱子移动，箱子跟随到机器人原位
- **拐弯规则**：拖拽只能沿直线，不能绕 90° 弯角
- **安全区规则**：普通箱子不能进入安全区（会"爆炸"），只有 R3 可以
- **挂接规则**：只有一个相邻箱子时自动挂接；多个相邻箱子时按 facing 方向选择
- **胜利条件**：R3 在 Repair Room（exit 格子）+ R1、R2 都在安全区 3×3 范围内

### 与标准 Sokoban 的关键区别

1. **有拉操作** → 大部分操作可逆，几乎不存在永久死局
2. **拐弯约束** → 拉不是万能的，走廊拐角处仍需绕路
3. **双机器人** → 状态空间含两个机器人位置，但也提供协作可能
4. **复杂度** → 仍然是 PSPACE-complete（Push-Pull Block Puzzles 的已证结论）

## 2. Solver 演进历史

### V1: 逐步 BFS（最初版本）

**思路**：每一步搜索是"一个机器人移动一格"。

**问题**：状态空间爆炸。90% 的搜索是机器人在迷宫里走路，不推箱子。Level 3（4 箱子）就超时。

**教训**：逐步搜索的粒度太细，机器人走路不应该是搜索的一部分。

### V2: 宏操作 A*（第一次重大改进）

**思路**：每次搜索转移 = "机器人走到箱子旁 + 推/拉一格"。走路用内部 BFS 判断可达性。

**改进**：搜索深度从几百步降到几十步。Level 3 能解了。

**残留问题**：每个推/拉只移一格，连续直线推 5 格 = 5 个搜索状态。

### V3: 多格滑动（宏操作增强）

**思路**：一次推/拉可以滑动 1~N 格，每个停靠距离是一个分支。

**改进**：减少搜索深度，展开时生成 attach → N 次 move → detach，不再每格都 attach/detach。

**讨论**：考虑过"只在交叉口停靠"的智能停靠点，但分析后认为在华容道式密集布局中不安全——中间位置的停靠可能是必要的。

### V4: 连通分量抽象（第二次重大改进）

**思路**：状态键中的机器人位置从精确坐标替换为连通分量 ID。

- 箱子把迷宫分割成若干连通分量
- 两个机器人各在某个分量中
- 同一分量内的所有位置对"能推哪些箱子"来说是等价的
- 精确位置存在侧表（witnessMap）中，用于走位 BFS 和动作展开

**改进**：机器人位置维度从 ~50² = 2500 缩减到 ~5² = 25，状态空间缩小约 100 倍。Level 6（5 箱子）从超时变为 119ms。

### V5: 组合数编码（编码优化）

**思路**：普通箱子位置排序后，用组合数 C(F, nNormal) 编码替代 F^nNormal 幂编码。

**改进**：编码空间缩小 ~140 倍（对 5 个普通箱子）。但实际搜索速度几乎不变——因为 Map 是哈希表，编码空间大小不影响查找速度。

**教训**：编码压缩对哈希表搜索的帮助有限。真正减少搜索量的是状态合并（连通分量抽象），不是编码紧凑性。

### V6: 非墙格子 ID 压缩

**思路**：只给非墙格子分配 0~F-1 的编号，M 从 140 降到 ~60。

**改进**：与组合数编码类似，主要是编码空间更紧凑，对实际搜索速度帮助有限。

## 3. 当前 Solver 架构（定稿版本）

### 状态定义

```
State Key = compSig × C(F, nNormal) × F + comboRank(sortedNormals) × F + r3FloorId
```

其中：
- `compSig = min(c1, c2) * 16 + max(c1, c2)`：两个机器人的连通分量 ID 排序后的签名
- `comboRank`：普通箱子位置的组合数编码
- `r3FloorId`：R3 的非墙格子编号
- 机器人视为无标签（交换 R1/R2 不影响状态键）

### 侧表

- `witnessMap: Map<key, packed(r1, r2)>`：每个状态的机器人精确位置
- `parent: Map<key, parentKey>`：搜索路径回溯
- `actionInfo: Map<key, packedAction>`：每个状态的操作信息（滑动距离+方向+机器人+push/pull）
- `gMap: Map<key, cost>`：到达每个状态的代价
- `lastMove: Map<key, packedBoxMove>`：反向剪枝用

### 搜索算法

加权 A*（权重 3）：
```
f = g + h
g = 走位步数 + 滑动步数（累计）
h = distToExit[R3] × 3
```

distToExit 是预计算的 BFS 迷宫距离表（只考虑墙，不考虑箱子）。

### 宏操作生成

每个状态展开时：
1. 计算连通分量（共享 compBuf，洪水填充）
2. 为两个机器人计算 BFS 可达性缓存（预分配 reachDistBufs）
3. 枚举：每个机器人 × 每个箱子 × 4 方向 × push/pull × 1~N 格滑动
4. 对每个合法后继：计算新连通分量 → 编码状态键 → 检查去重 → 加入堆

### 剪枝

- **状态去重**：`parent.has(nk)` 拦截已访问状态
- **反向操作剪枝**：不允许立即撤销上一步（同一箱子、反方向，不区分 push/pull）
- **安全区限制**：普通箱子不能进入安全区
- **连通分量合并**：不同机器人精确位置但同一分量 → 同一状态键

### 解的展开与验证（expandAndValidate）

搜索找到目标状态后：
1. 沿 parent 链回溯，得到状态键序列
2. 从 witnessMap 取每一步的精确机器人位置
3. 通过比较相邻状态的箱子位置集合，识别哪个箱子移动了（处理 canonical 排序的影响）
4. 生成真实游戏动作序列：selectRobot → 走位（BFS）→ toggleAttach → move × N → toggleAttach
5. **每个动作通过 applyAction 在游戏状态克隆上验证**
6. 验证失败 → 丢弃这个解，搜索继续寻找下一个目标状态

## 4. 踩过的坑

### 坑 1: 纯箱子搜索不含机器人位置

**做了什么**：第一版宏操作搜索只搜索箱子配置，不追踪机器人位置。找到箱子方案后再"展开"成机器人动作。

**后果**：大量"箱子方案"在展开时发现机器人根本走不到推箱子的位置。Level 1 都解不了。

**教训**：推箱子游戏不是纯箱子谜题。机器人能不能到位是状态的一部分，必须在搜索时就验证。

**修复**：把机器人位置（后来升级为连通分量）纳入搜索状态。

### 坑 2: Canonical 编码下的箱子识别错误

**做了什么**：普通箱子位置排序后编码（消除对称性）。展开解时，通过逐索引比较 `bp[i] !== bpNext[i]` 来识别哪个箱子移动了。

**后果**：排序导致箱子索引变化。例如 [30, 50] 中 50 移到 10，排序后变成 [10, 30]。逐索引比较认为是位置 30 的箱子变了，但实际是位置 50 的箱子移到了 10。机器人走到了错误的箱子旁边，操作了错误的箱子。

**教训**：canonical 编码丢失了箱子身份。识别移动的箱子必须比较**位置集合**，而非逐索引比较。

**修复**：改为集合差异检测——找出旧集合中有但新集合中没有的位置（oldPos），和新集合中有但旧集合中没有的位置（newPos）。

### 坑 3: 共享缓冲区的"bug"不是 bug

**做了什么**：`computeComponents` 使用共享的 `compBuf`。我认为后继状态的 `computeComponents` 调用会覆盖当前状态的 `comp`，造成数据错误。于是改为每次分配新的 `Uint16Array`。

**后果**：性能大幅下降（大量 GC 分配），且后来发现"bug"根本不存在——共享缓冲区被覆盖的值在后续代码中不再使用。

**教训**：
1. 不要假设"看起来像 bug"的代码就是 bug。先验证是否真的导致了错误结果。
2. K=0 测试是验证这种问题的好方法——如果 K=0 的结果和没有改动的版本完全一致，说明改动本身引入了差异。
3. 每次修改前备份，确保能精确对比。

### 坑 4: 启发函数改进反而退化

**做了什么**：三次尝试改进启发函数：
1. Dijkstra（K=20）：从 R3 到 exit 做 Dijkstra，箱子格代价 21
2. 0-1 BFS blocker count × 10：数 R3 到 exit 最短路径上的箱子数
3. blocker count × 0.1（作为 tie-breaker）

**后果**：
- Dijkstra K=20：L3 大幅改善（8097→1550），但 L6/L7 大幅退化（L7: 6226→26434）
- blocker × 10：L1/L4 改善，但 L3/L7 退化
- blocker × 0.1：L1-L6 微改善，L7 仍然退化（6226→26539）

**根因分析**（参考 ChatGPT 咨询结论）：
- blocker count 分辨率太低（通常只有 0/1/2），作为启发值加法项时充当了"路由偏差"而非精确估计
- 加权 A* 放大了不可靠的次级信号，导致搜索被引向"看起来干净但实际更远"的路径
- L7 对 tie-breaking 极其敏感——大量状态有相同的 f 值，微小扰动就导致完全不同的搜索路径

**ChatGPT 的建议**：blocker 信息应该作为 tie-breaking 的多级排序标准，而不是加到 f 值里。但我们实测 K=0.1 仍然让 L7 退化，说明即使很小的扰动也会在 L7 的大量 tie 中引起级联效应。

**教训**：
1. 启发函数改进必须在**所有关卡**上测试，不能只看单一关卡的改善
2. "更聪明"的启发不一定更好——如果信号不可靠，不如不加
3. 用 K=0 测试验证"改动是否引入了非启发性的差异"
4. 保留测评脚本（test-solver.ts），每次改动后立即跑全套回归测试

### 坑 5: 目标检查中 R3 占据 exit 格子

**做了什么**：目标检查改为 `comp[exitIdx]` 来判断机器人能否到达安全区。

**后果**：R3 在 exit 上时，exit 格子被标记为"有箱子"，`computeComponents` 给它分配 compId=0（未访问）。`comp[exitIdx] === 0` 而机器人的分量 ≥ 1，所以永远 false → 找不到解。

**教训**：R3 是一个箱子。当它到达 exit 时，exit 格子在连通分量计算中被视为"被箱子占据"而跳过。目标检查必须查安全区的**邻近格子**的分量，而非 exit 本身。

**修复**：恢复为遍历安全区所有格子，检查是否有任何安全区格子和机器人在同一分量。

### 坑 6: 不同代码版本产生不同的搜索路径

**现象**：从旧版 solver 的基础上重写启发函数部分，即使设 K=0（数值上应该和旧版完全一致），状态数也不同（L1: 551 vs 1250）。

**根因**：虽然启发公式的数值相同，但代码结构的微小差异（变量命名、中间表达式的求值顺序等）可能影响了堆中 tie-breaking 的行为。

**教训**：
1. 要在旧版代码上做**最小改动**，而非重写后试图对齐
2. K=0 是验证"是否引入了非期望差异"的金标准
3. 用 `md5` 对比编译产物来精确验证版本一致性

### 坑 7: expandAndValidate 缺少"机器人走回安全区"步骤

**现象**：搜索找到了完美的 6-macro-move 解（零箱子操作，g=40），但 expandAndValidate 返回 null。搜索被迫继续探索，最终找到包含 30 步无用箱子操作的次优解。Hard 关卡从理论 10ms 退化到 8+ 秒。

**根因**：搜索用连通分量抽象判定目标——"R3 在 exit 且两个机器人的分量包含安全区格子"。但 expandAndValidate 执行完 macro moves 后，R1 的 witness 在迷宫中（如 (2,7)），虽然和安全区在同一连通分量，但游戏的 checkWin 要求机器人**站在安全区格子上**。`s.won` 为 false。

**修复**：在 expandAndValidate 的 macro moves 执行完后，对每个不在安全区的机器人，BFS 找最近安全区格子并走过去。

**影响**：修复后 Hard 关卡从 TIMEOUT 降到 10ms（1000× 加速），L5 从 507 states 降到 30 states，L9 从 10446 降到 4795。之前所有"expandAndValidate 失败后搜索继续找次优解"的现象全部消失。

## 4.5 expandAndValidate 理论失败场景

expandAndValidate 将搜索链的 macro moves 展开为真实游戏动作。以下情况可能导致失败（返回 null）：

### 场景 1: 走位不可达（walkRobotTo 返回 false）

**触发条件**：搜索用连通分量判断"机器人能到达推/拉位置"，但 expand 用精确位置 BFS。如果另一个机器人恰好挡在唯一通路上，BFS 绕不过去。

**详细机制**：
- 搜索中，两个机器人在同一连通分量被视为等价（无标签）
- witness 记录的是第一次到达该 abstract state 时的精确位置
- 后续 macro move 的 walkRobotTo 用 witness 位置做 BFS，把另一个机器人当障碍物
- 如果 witness 位置恰好让一个机器人挡住另一个的唯一通路 → 走位失败

**发生概率**：低。大多数地图有多条通路。在极窄走廊（宽 1）中两个机器人面对面时可能触发。

**缓解**：搜索继续，找到不同 witness 的替代路径。代价是多搜一些状态。

### 场景 2: toggleAttach 挂错箱子

**触发条件**：机器人旁边有多个箱子，facing 方向不指向目标箱子。

**详细机制**：
- 游戏规则：多箱子相邻时，toggleAttach 按 facing 方向选择
- expand 在 pull 前会尝试调整朝向（退一步再进一步）
- 如果调整朝向的位置被墙/箱子/机器人堵住 → 无法调整 → 可能挂到错误箱子

**验证**：expand 在 attach 后检查 `attached.pos === 目标箱子位置`，不匹配则 detach + 返回 null。

**发生概率**：极低。只在极度拥挤的布局（多个箱子紧密排列 + 无调整空间）中可能触发。

### 场景 3: push/pull 被游戏规则拒绝

**可能的子情况**：
- **拐弯规则**：robot.facing 和 movement direction 不符合直线拖拽约束。理论上不应发生，因为 pull 前已设置方向。
- **安全区限制**：普通箱子被意外推入安全区。搜索阶段已检查此约束，但 expand 的走位 BFS 可能让机器人经过意外路径推动了其他箱子。
- **碰撞**：两个机器人在同一格。搜索阶段不追踪精确位置碰撞。

**发生概率**：极低。走位 BFS 避开所有箱子和另一个机器人。

### 场景 4: 搜索链的 box-move 检测失败（oldPos < 0）

**触发条件**：canonical 编码解码后，比较前后状态的箱子位置集合，找不到差异。

**可能原因**：
- canonical 排序导致的数值巧合（极罕见）
- 搜索链中有状态键重复但 box 布局不同（不应发生）

**发生概率**：理论上不可能，但作为防御性检查保留。

### 当前状态（2026-04-16）

修复"走回安全区"bug 后，10 个测试关卡（含 Hard）全部 **零 expand 失败**。搜索找到的第一条路径一次性展开成功。

## 5. 启发函数的现状与结论

### 当前启发函数（定稿）

```
每个状态展开时：
  从 R3 位置出发做 Dijkstra（空格=1，普通箱子=1+K=11，墙=不通）
  早停：找到 R1、R2、exit 三个目标后停止

h_raw = dist[R1] * 1.0 + dist[R2] * 1.0 + dist[exit]
f = g + h_raw * 3
如果本次操作移动了 R3：f 再减 1
```

参数：K=10, W=3, w_robot=1.0

### 关键发现

1. **w_robot 在修复后无影响**：w=0.5/0.7/1.0 产生完全一样的结果。之前观察到的差异全部来自 expand bug。连通分量抽象下，robot 的精确距离权重被分量合并掩盖了。

2. **W 越大，Hard（空路径）越差**：W=5 时 Hard 暴涨到 17376（真实的搜索行为，非 bug）。高 W 在大量相同 f 值的状态中缺乏区分力。

3. **K 越大，L9（密集腾挪）越差**：K=20 时 L9 从 4795 升到 9290。过高的箱子惩罚导致搜索过度回避箱子。

4. **没有单一参数组合在所有关卡上最优**。W=3 K=10 是最稳定的选择——没有任何关卡严重退化。

### 什么有效

- `distToExit[R3] × 3`：简单、快速、稳定。在所有关卡上表现一致。
- R3 优先 tie-breaking（`- (bi === redIdx ? 1 : 0)`）：微小偏好，稳定有效。

### 什么无效

- 任何加到 f 值里的 blocker 惩罚项：帮一些关卡，伤另一些。
- Dijkstra 路径代价：计算太贵，且高 K 值导致过度回避 blocker。
- 随机 tie-breaking：方差太大，不可靠。

### 专家建议（未实现）

ChatGPT 建议的 lexicographic tie-breaking（多级排序）：
```
1. g + 3 * distToExit[R3]          ← 主排序
2. distToExit[R3]                   ← 同 f 时优先更近的
3. mandatoryBlockers                ← 门口走廊里的箱子数
4. lastMoveAffectedR3 ? 0 : 1       ← 优先推 R3
5. -g                               ← 同 f 时优先搜得更深的
```

理论上合理，但需要改 MinHeap 支持多级比较。考虑到当前性能已经足够，暂未实现。

## 7. 不能安全剪枝的情况

以下剪枝方案经过分析后**放弃**，因为可能丢失有效解：

| 方案 | 为什么不安全 |
|------|------------|
| 间隔撤销（A→B→A 回来） | A 的临时移动可能是让机器人绕路到 B 的必要条件 |
| "不相关"箱子硬剪枝 | 移动它可能连通两个分量，改变可达性 |
| 走廊中间停靠跳过 | 华容道式密集布局中，中间位置可能是关键 |
| 多格滑动的部分撤销 | 当前反向剪枝已覆盖（同一箱子立即反方向全部拦住） |

## 7.5 已知问题与未来改进方向

### compBuf 幽灵状态——已通过 safeFlags 解决

`computeComponents` 使用共享的 `compBuf`。push/pull 循环中每个后继调用 `computeComponents` 会覆盖 `compBuf`。随后的 walkOnly 读到的是被覆盖的错误分量数据，产生错误的 compSig，从而生成"幽灵状态"。

**根因分析**：walkOnly 转移的真正含义是"机器人回到安全区"。这改变了安全区内机器人数量，但原始 state key 没有编码这个信息。compBuf 被覆盖恰好让 walkOnly 产生不同的 key，间接区分了"0/1/2 个机器人在安全区"的情况。

**解决方案**：引入 `safeFlags`——安全区内机器人的无标号计数（0、1、2），加入 state key。公式变为：

```
key = (compSig * 3 + safeFlags) * comboSize * F + cr * F + r3fid
```

同时修正 walkOnly 部分，用 `const walkComp = computeComponents(bp)` 独立计算分量，不再依赖被覆盖的 `compBuf`。

safeFlags 使用无标号计数（而非区分 R1/R2）以保持与连通分量抽象的一致性——分量签名不区分具体是哪个机器人在哪个位置，safeFlags 同样只关心"几个"在安全区。

### BFS 可达性：另一个机器人作为障碍物（有意保留）

搜索的 BFS 可达性缓存中，另一个机器人**被当作障碍物**。这与连通分量抽象存在理论上的不一致——分量不区分机器人互让，但 BFS 认为对方挡路。

**保留原因**：BFS 走步数直接用于 g 值（搜索代价）。如果 BFS 忽略另一个机器人，走步数会偏少（穿越了实际无法穿越的位置），导致 g 低估。实测中去除此限制后 L3/L4/L6 超时——不是因为状态爆炸，而是因为 g 低估导致搜索方向偏移。

**权衡**：g 精度 > 状态完备性。当前 BFS 的保守估计让搜索更聚焦，虽然偶尔漏掉需要互让的路径，但整体性能远优于无限制版本。

### bestF 方案（理论正确，实践待调优）

**思路**：用 f 值（而非 parent.has）决定是否接受新的搜索路径。

```
当前: if (!parent.has(nk)) { 存入 }  ← 只保留第一个到达的
改为:
  expanded = Set()      // 已展开的状态
  bestF = Map()         // 每个状态见过的最小 f

  入堆条件: !expanded.has(nk) && (!bestF.has(nk) || bestF.get(nk) > newF)
  出堆条件: if (expanded.has(ck)) continue; expanded.add(ck);
```

**核心问题**：连通分量抽象下，同一 key 的不同 witness（机器人精确位置）可以有完全不同的 Dijkstra 启发值。搜索只保留第一个到达的 witness（`parent.has` 过滤），如果第一个 witness 的 h 不好，搜索就走偏了。bestF 让后续更优 witness 有机会替换。

**实测结果**：L8/L9 改善，但 L1/L3/L4/L6 大幅退化（状态数暴增 10-100 倍）。原因是过多的 f 改善触发重新入堆，堆中条目爆炸。

**改进方向**：
- 只在 f 改善超过阈值（如 >5）时才重新入堆
- 限制每个 state key 的最大重新入堆次数
- 只对 walkOnly 转移应用 bestF（不影响 push/pull 的去重逻辑）

此方案保留为未来优化方向，当前不实施。

### R2 来回出差问题

搜索链中 R2 反复在安全区和迷宫之间来回（L8 有 4 次、L9 有 4 次来回）。原因是搜索链包含 walkOnly 转移和交替的 push/pull 操作，导致 R2 被反复调用。

**切换惩罚**（搜索中对换手操作加 f 惩罚）已试验但效果不佳——必要的双机器人协作被惩罚，反而变慢。

**expand 阶段换机器人**也已试验但导致搜索链预期与实际不一致，大量 expand 失败。

此问题留作解后处理优化：在 expand 完成后，扫描并消除不必要的来回行程。

## 8. 搜索确定性的验证方法

经验证，当前 solver 是**完全确定性的**——相同输入产生相同输出。

### 验证手段

1. **K=0 测试**：将 blocker 权重设为 0，确认状态数与无 blocker 版本完全一致
2. **MD5 对比**：`npx vite build --base='./'` 后对比编译产物的 MD5 hash
3. **test-solver.ts**：固定的 7 关回归测试，比较状态数 + 动作数 + 验证结果
4. **多次运行**：确认结果不变（排除随机性）

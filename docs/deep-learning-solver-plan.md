# Deep Learning Solver Plan: Maze Trials

## 目标

用深度学习训练一个"棋感"网络，替代手写启发函数，让 AI solver 具备：
- 更精准的状态评估（哪些局面容易解，哪些难）
- 泛化能力（在从未见过的地图上也能高效求解）
- 可展示的"学习与进化"能力（对标 AlphaZero 的叙事）

---

## 架构概览

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  地图生成器   │────▶│  数据标注管线  │────▶│  训练管线     │
│ (随机关卡)   │     │ (solver求解)  │     │ (CNN/GNN)    │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                                                  ▼
┌──────────────┐                          ┌──────────────┐
│  浏览器 UI   │◀──HTTP/WS──▶             │  训练好的模型  │
│ (渲染+操作)  │             │             └──────┬───────┘
└──────────────┘             │                    │
                             ▼                    ▼
                      ┌──────────────────────────────┐
                      │     Mac Mini 本地服务          │
                      │  Python solver + PyTorch 推理  │
                      │  A* 搜索 + NN 启发函数         │
                      └──────────────────────────────┘
```

**关键决策：推理在 Mac Mini 本地运行，不在浏览器端。**
- 浏览器只负责渲染和用户交互
- AI SOLVE 请求通过 HTTP/WebSocket 发到本地 Python 服务
- 无需模型导出、量化、浏览器端 ML 框架
- 模型大小和推理速度限制大幅放宽

## Phase 1: 数据生成

### 1.1 随机地图生成器

自动生成大量可解的关卡：

- 地图大小：7×7 到 10×10（含安全区）
- 箱子数量：2-6 个普通箱子 + R3
- 墙壁密度：随机但保证连通性
- 安全区 + 窄门：固定结构（底部 3×3 + 1 格门）
- R3 起始位置：随机但保证有解
- 生成后用现有 solver 验证可解性，不可解的丢弃

**产出**：数万到数十万组 (地图, 箱子初始布局) 对

### 1.2 数据标注

对每个地图，用现有 solver 求解，记录：

**训练数据格式**：
```
{
  map: 2D grid (walls/floor/exit/safe),
  state: {r3_pos, normal_box_positions, robot_positions},
  label: {
    solvable: boolean,
    optimal_cost: number,      // solver 找到的解的步数
    states_explored: number,   // 搜索量（难度指标）
  }
}
```

**增强数据**：对每个求解过程中的中间状态也标注"剩余步数"，这样一次求解可以产生几十到几百个训练样本。

### 1.3 数据量估计

- 生成 50,000 个关卡
- 每个关卡平均 100 个中间状态
- 总计约 500 万训练样本
- 存储：每个样本 ~200 字节 → 约 1 GB

## Phase 2: 网络设计

### 2.1 输入表示

将游戏状态编码为多通道 2D 图像（类似 AlphaZero 的棋盘表示）：

```
通道 0: 墙壁 (binary)
通道 1: 地板 (binary)
通道 2: 安全区 (binary)
通道 3: 出口 (binary)
通道 4: R3 位置 (binary)
通道 5: 普通箱子位置 (binary)
通道 6: 机器人 1 位置 (binary)
通道 7: 机器人 2 位置 (binary)
通道 8: distToExit 归一化 (float, 预计算)
```

输入形状：`(9, H, W)` — 最大 `(9, 14, 10)`，padding 到固定大小 `(9, 16, 16)`

### 2.2 网络架构

**方案 A: CNN（简单，适合起步）**

```
Input (9, 16, 16)
  → Conv2D(32, 3×3, padding=same) + ReLU
  → Conv2D(64, 3×3, padding=same) + ReLU
  → Conv2D(64, 3×3, padding=same) + ReLU
  → ResBlock × 4 (64 filters)
  → GlobalAvgPool
  → Dense(128) + ReLU
  → Dense(1)  ← 预测剩余步数
```

参数量：~100K，推理速度快

**方案 B: GNN（更适合图结构）**

将地图建模为图（节点=格子，边=相邻关系），用 Graph Attention Network：
- 节点特征：该格子的类型 + 是否有箱子/机器人
- 边特征：方向（上下左右）
- 优点：天然适配不同大小的地图，不需要 padding
- 缺点：实现稍复杂

**推荐从方案 A 起步**，效果不够再升级到 B。在 Mac Mini 上跑 PyTorch，两种方案都无性能问题。

### 2.3 训练目标

**Value Network（回归）**：
- 输入：当前游戏状态
- 输出：预测剩余步数（scalar）
- 损失函数：MSE 或 Huber Loss
- 这个值直接替代 `distToExit[R3] * 3` 作为启发函数 h

**可选 Policy Network（分类）**：
- 输出：下一步最佳操作的概率分布
- 用于指导搜索顺序（类似 AlphaZero 的 policy head）
- 比 value network 更难训练，建议第二阶段做

## Phase 3: 训练

### 3.1 训练环境

- 框架：PyTorch（训练端）
- 硬件：单 GPU 足够（模型很小）
- 训练时间估计：几小时到一天

### 3.2 训练策略

**Stage 1: 监督学习**
- 用 solver 产生的 (状态, 剩余步数) 对直接训练
- 目标：网络学会"哪些局面离解近，哪些远"
- 验证：在测试集上看 MAE（平均绝对误差）

**Stage 2: 自我进化（可选）**
- 用 Stage 1 的网络作为启发函数跑 solver
- 记录搜索过程中发现的更好路径
- 用新数据微调网络
- 重复 → 网络越来越准，solver 越来越快
- 这就是"自我进化"的展示点

### 3.3 评估指标

- 搜索效率：同一关卡，用 NN 启发 vs 用 distToExit 启发，比较探索状态数
- 泛化能力：在训练时没见过的地图上的表现
- 推理速度：每次 NN 评估的耗时（必须 < 0.1ms 才实用）

## Phase 4: 部署到 Mac Mini 本地服务

### 4.1 架构

```
浏览器                    Mac Mini
  │                         │
  │  POST /solve            │
  │  {level_ascii}  ──────▶ │  Python FastAPI 服务
  │                         │  ├─ 解析关卡
  │                         │  ├─ A* 搜索 + PyTorch NN 启发
  │                         │  └─ 返回动作序列
  │  ◀────────────────────  │
  │  {actions: [...]}       │
  │                         │
  │  按序回放动作            │
```

- **不需要模型导出或量化**——直接用 PyTorch 跑
- **不需要 ONNX/TFJS**——纯 Python 服务端
- 浏览器只需要 fetch 一次，拿到完整动作序列后本地回放

### 4.2 Python Solver

将现有 TypeScript solver 的核心逻辑移植到 Python：
- 宏操作 A* 搜索框架
- 连通分量抽象
- 组合数编码
- expandAndValidate

**启发函数替换**：
```python
# 旧：
h = dist_to_exit[r3_pos] * 3

# 新：
state_tensor = encode_state(boxes, robot_components)
h = value_network(state_tensor).item()
```

### 4.3 推理速度

- Mac Mini M2/M4 CPU：PyTorch CNN 100K 参数，单次推理 ~0.02ms
- Mac Mini MPS (Apple GPU)：可进一步加速到 ~0.005ms
- 搜索 10000 个状态 × 0.02ms = 0.2 秒，完全可接受
- **模型可以更大**（500K-1M 参数），不受浏览器限制

### 4.4 混合策略

不完全依赖 NN——用 NN + 传统启发的加权组合：

```python
h = alpha * nn_value + (1 - alpha) * dist_to_exit[r3] * 3
```

alpha 从 0 渐变到 1，可以展示"AI 逐渐学会依赖自己的判断"的进化过程。

### 4.5 保留浏览器端 Solver 作为后备

现有的 TypeScript solver 保留在浏览器端：
- 网络不通时可以本地求解
- 小关卡（L1-L5）本地就够快
- 大关卡或需要 NN 启发时才走 Mac Mini

## 风险与备选方案

### ~~风险 1: NN 推理太慢~~
在 Mac Mini 上跑 PyTorch，推理速度不再是瓶颈。即使 1M 参数模型也能 < 0.1ms/次。

### 风险 1: 泛化能力不够
- 备选：增加训练数据多样性（更多地图变体）
- 或：用 GNN 替代 CNN 获得更好的结构泛化

### 风险 2: 训练数据生成太慢（solver 求解慢）
- 备选：只用小地图（7×7, 8×8）训练，大地图用迁移学习
- 或：用次优解（设 solver 时间上限 1 秒，超时的标注为"难"）

---

## 与现有架构的兼容性

**完全兼容。** NN 只替换启发函数 h 的计算方式，搜索框架（宏操作 A*、连通分量抽象、组合编码、expandAndValidate）全部保留。这意味着：

- 可以随时在 NN 启发和传统启发之间切换
- NN 出错时 expandAndValidate 兜底保证解的正确性
- 不需要重写任何游戏逻辑或搜索框架
